# Test Finalization Reference

## Purpose

Incorporate reviewer feedback into generated tests, run a comprehensive compliance check against ALL skill rules, and produce final polished, PR-ready test code.

## What to Fix

- **Every issue flagged** in the review report -- no exceptions
- **Add missing tests** identified by the reviewer
- **Improve assertion quality** where flagged LOW or MEDIUM
- **Fix naming** to follow `MethodName_Scenario_ExpectedBehavior`
- **Match the repository's framework**: Keep the existing test framework/mocking stack and APIs used by the repo.
- **For C# projects already on MSTest+Moq**: keep that stack and apply MSTest-version-compatible APIs.
- **Ensure independence** -- no test depends on another or shared mutable state
- **Ensure determinism** -- remove time/random/network dependence
- **Replace tautological tests** with proper behavioral assertions
- **Add plan ID comments** mapping each test to its test plan ID (recommended, not blocking)
- **Preserve all traceability metadata** -- `[TestCategory("AutoGenerated")]`, `[TestCategory("CopilotSkill:autocoverage")]`, `[TestProperty]` attributes, and the header comment block
- **Remove tests violating exclusion rules** -- any test that tests constants, DTOs, constructors (simple), properties, default values, or logging conventions must be removed, not fixed

## Final Compliance Checklist (MANDATORY)

Before delivering the final code, run through EVERY check below. If any check fails, fix it or remove the offending test. Do NOT deliver code that violates these rules.

### Exclusion Rule Compliance

Verify NO test targets any of the following. If found, REMOVE the test entirely (do not attempt to fix it):

| Check | How to Verify |
|-------|--------------|
| Constants / constant fields | Does the test assert on a compile-time constant value? Remove. |
| Default value assignments | Does the test assert a field equals its default? Remove. |
| Auto-properties (get/set only) | Does the test set a property then assert it back? Remove. |
| DTO/POCO classes | Does the test only exercise a class with no logic (properties only)? Remove. |
| Logging conventions | Does the test verify log output, structured logging templates, or log categories? Remove. |
| Simple constructors | Does the test only verify that constructor assigns parameters to fields (no branching)? Remove. |
| Generated code | Does the test target auto-generated files (designer.cs, .g.cs, etc.)? Remove. |
| Code excluded by `.runsettings` | Does the test target code in the CI coverage exclusion list? Remove. |

### No-Refactoring Compliance

Verify the finalization step itself did NOT introduce any of these prohibited changes:

| Check | How to Verify |
|-------|--------------|
| No folder renames | Were any folders renamed or moved? |
| No file moves | Were any source files moved to different directories? |
| No project file renames | Were any .csproj/.vbproj files renamed? |
| No assembly name changes | Was any assembly name changed? |
| No namespace changes in production code | Were namespaces changed in any non-test file? |
| No project layout changes | Was the project structure reorganized? |
| No production package changes | Were package references added/removed in production projects? |
| No production code changes | Was ANY production code file modified? (Test files only) |

### No Co-Located Projects

| Check | How to Verify |
|-------|--------------|
| Test .csproj in separate folder | Is the test .csproj in a different directory from the source .csproj? |
| Follows repo convention | Does the test project location match where other test projects live? || No duplicate test project for same source | Does another test project already reference the same source project (even under a different name)? If yes, add tests there instead. |
### Test Quality Compliance

| Check | How to Verify |
|-------|--------------|
| No tautological assertions | Does any test re-compute expected values using production logic? Remove/fix. |
| No assert-free tests | Does any test call a method without asserting observable behavior? Fix or remove. |
| No copy-paste production logic | Does any test duplicate production code to derive expected values? Fix. |
| No testing-the-mock | Does any test assert on a mock return value that was explicitly set up? Fix. |
| No over-mocking | Is the class under test itself mocked? Are POCOs mocked? Fix. |
| No implementation coupling | Does any test verify internal call order instead of outputs? Fix. |
| No magic numbers | Are expected values unexplained? Add comments or named constants. |
| No excessive setup | Does any test have 50+ lines of arrange for one assert? Simplify. |
| No reflection for private methods | Does any test use reflection to access private members? Fix to use public API. |
| No time/random dependence | Does any test use DateTime.Now, Random, Guid.NewGuid()? Fix. |
| No real external calls | Does any test call real services, databases, network, file system? Mock them. |
| Assertion density >= 1.5 avg | Count total asserts / total tests. Below 1.5? Add assertions. |
| One test class per file | Does any file contain multiple unrelated test classes? Split into separate files. |
| File naming matches source | Is the test file named `<SourceClass>Tests.cs`? Rename if not. |
| No behavioral duplicates | Does any test duplicate a scenario already covered by an existing test? Remove. |

### Framework Compliance (C#)

| Check | How to Verify |
|-------|--------------|
| Uses MSTest, not xUnit/NUnit | `[TestClass]`/`[TestMethod]`, not `[Fact]`/`[Theory]` or `[Test]` |
| Uses Moq, not FluentAssertions | `Assert.AreEqual`, not `Should().Be()` |
| Correct MSTest version APIs | v4+: `Assert.ThrowsExactly<T>()`, `[TestMethod]`+`[DataRow]`; no `[DataTestMethod]`, no `[ExpectedException]` |
| No obsolete API usage | No calls to methods marked `[Obsolete]` |
| Zero compiler warnings | Build produces no warnings (critical if `TreatWarningsAsErrors` is enabled) |

### Traceability Compliance

| Check | How to Verify |
|-------|--------------|
| `[TestCategory("AutoGenerated")]` on every test class | Grep for `[TestClass]` without `[TestCategory("AutoGenerated")]` |
| `[TestCategory("CopilotSkill:autocoverage")]` on every test class | Same |
| Header comment block with generator name, date, model, source file | Check top of each test file |
| Plan ID comments on each test (recommended) | `// TP-<SourceFile>-<ShortName>` comment where practical — missing comments do not block compliance |

### Lint / Coding Standards

| Check | How to Verify |
|-------|--------------|
| Matches repo formatting | Indentation, line length, brace placement match existing test files |
| Passes linter/formatter | Run the lint command from Stage 0 -- zero new violations |
| Import/using conventions | Same order and grouping as existing test files |
| No `#pragma warning disable` | Unless source code itself uses them |

## Process

1. Read the review report.
2. Read the original generated test code.
3. Read the source code under test.
4. Read the test plan for reference.
5. Apply all fixes from the review.
6. **Run the Final Compliance Checklist above** -- every section, every row.
7. Remove any test that fails a compliance check and cannot be fixed.
8. After fixing, re-run build + test verification (Stage 6 in the pipeline).

## Output

Produce the final test code -- complete, compilable, with all imports. Copy-paste ready for a PR.

After the code, include:

```markdown
### Changes from Review
- **Fixed**: <list of fixes applied>
- **Added**: <list of new tests added>
- **Removed/Replaced**: <list of tests removed or replaced>

### Compliance Checklist Result
- Exclusion rules: PASS (N tests checked, 0 violations)
- No-refactoring: PASS (no production code changes)
- No co-located projects: PASS
- Test quality: PASS (N tests, 0 anti-patterns)
- Framework compliance: PASS
- Traceability: PASS
- Lint/standards: PASS
```

## Rules

- If review says all tests are approved, still run the Final Compliance Checklist before delivering.
- If a test is marked REJECT, replace it entirely with a correct implementation -- do not just patch it.
- If a test violates exclusion rules, remove it entirely -- do not try to fix it.
- If the compliance checklist reveals issues not caught in review, fix them now. Do not deliver non-compliant code.
